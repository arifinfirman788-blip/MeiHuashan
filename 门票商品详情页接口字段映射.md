# 门票商品详情页接口字段映射

## 1. 文档说明

本文档以当前页面文件为准：

- Vue 页面：`src/pages/TicketDetailPage.vue`
- 静态预览：`门票商品详情页.html`
- 接口文档：黔云联创《商城小程序 V2 接口文档》
- 接口基础路径：`/api/v2/mallapi`
- 测试环境前缀：`https://api.mall.chain-idea.com`

映射状态说明：

| 状态 | 含义 |
|---|---|
| 已对应 | 页面字段可以直接由接口字段赋值 |
| 组合/派生 | 需要由多个接口字段组合，或由接口数据计算得到 |
| 无直接对应 | 页面交互、固定文案或客户端状态，黔云联创接口没有对应字段 |
| 当前缺字段 | 接口要求或支持，但当前页面没有对应控件/数据字段 |
| 待黔云联创确认 | 文档有候选字段，但业务语义或取值规则仍需黔云联创确认 |

## 2. 页面使用的接口

| 页面用途 | 接口 | 页面使用范围 |
|---|---|---|
| 商家/景区基础信息 | `GET /api/v2/mallapi/app/merchant/getMerchant` | 景区地址、营业时间、商家图片、商家介绍、景区名称 |
| 票型列表 | `GET /api/v2/mallapi/app/resources/getResourcesByResourcesGroup` | 资源名称、价格、库存状态、简短介绍、标签、保险标识 |
| 票务资源详情 | `GET /api/v2/mallapi/app/resources/getResourcesMoreInfo` | 资源名称、资源图片、核验方式、实名制、SKU、标签、资源介绍 |
| 日期/价格/库存/时段 | `GET /api/v2/mallapi/app/resources/getResourcesStockAndPrice` | 游玩日期、票型价格、停售状态、库存余量、入园时段、有效期 |
| 可选保险 | `GET /api/v2/mallapi/app/resources/getOptionalTypeResourcesList` | 保险名称、价格、是否必买、保险介绍 |
| 创建订单 | `POST /api/v2/mallapi/app/order/createOrder` | 联系人、票型、SKU、数量、日期、时段、游客信息 |
| 微信支付 | `POST /api/v2/mallapi/wechat/pay/wechatPayment` | 获取微信支付参数、确认免密/0 元支付结果 |
| 订单详情 | `GET /api/v2/mallapi/app/order/getOrderByNo` | 订单金额、联系人、游客、订单状态 |
| 核销码 | `GET /api/v2/mallapi/app/order/getResourcesOrderVoucher` | 核销码、核销码图片、核销状态、游客与票码关联 |
| 可用证件类型 | `GET /api/v2/mallapi/app/baseinfo/getCardTypes` | 证件类型选择、默认证件类型 |

## 5. 微信支付调用与订单状态逻辑

### 5.1 正式调用顺序

```text
填写订单信息
  ↓
前端校验联系人、游客、日期、时段和票型数量
  ↓
createOrder 创建订单
  ↓
读取 orderNo、orderId、payAmount、orderStatus
  ↓
payAmount = 0 或接口返回 paySuccess=true？──是──→ 查询订单状态
  │
  否
  ↓
wechatPayment 获取 payInfo
  ↓
wx.requestPayment 调起微信支付
  ↓
支付成功/取消/失败
  ↓
getOrderByNo 查询黔云联创订单最终状态
  ↓
支付结果页按订单状态展示
```

### 5.2 创建订单后调用微信支付

第一步调用：`POST /api/v2/mallapi/app/order/createOrder`

| 处理项 | 使用字段 | 处理规则 |
|---|---|---|
| 订单编号 | 返回 `orderNo` | 作为后续支付请求的商户订单号来源，不使用前端临时生成的订单号 |
| 订单主键 | 返回 `orderId` | 保存到订单上下文，查看订单时按接口要求使用 |
| 应付金额 | 返回 `payAmount` | 支付结果页展示服务端金额，不使用前端预估金额覆盖 |
| 初始状态 | 返回 `orderStatus` | 先按状态归一化；状态值含义待黔云联创确认 |

第二步调用：`POST /api/v2/mallapi/wechat/pay/wechatPayment`

请求字段：

| 字段 | 来源 | 规则 |
|---|---|---|
| `mchOrderNo` | `createOrder.orderNo` | 传创建订单返回的真实订单号 |
| `openid` | 当前微信用户身份 | 从小程序登录态获取，不由用户手工填写 |
| `appId` | 小程序配置 | 使用当前小程序的 AppID，不写死在页面代码中 |

返回字段：

| 字段 | 处理规则 |
|---|---|
| `paySuccess` | `true` 时表示接口已完成免密/0 元支付结果确认；仍建议查询订单状态后展示结果 |
| `payInfo` | 非 0 元支付时，按微信 SDK 要求转换后传入 `wx.requestPayment`；具体内部字段以黔云联创返回为准 |
| `mchOrderNo` | 用于校验支付请求对应的订单号，必须与创建订单返回的订单号一致 |

第三步调用微信小程序支付：

```js
wx.requestPayment({
  ...payInfo,
  success: () => queryOrderStatus(orderNo),
  fail: error => handlePaymentFailure(error, orderNo)
})
```

`wx.requestPayment` 的成功回调只代表用户端支付调用成功，不能直接等同于出票成功；必须继续调用 `GET /api/v2/mallapi/app/order/getOrderByNo` 查询订单状态。

### 5.3 订单状态归一化

页面不直接把接口状态值写死在模板中，统一通过 `orderStatus` 转换为以下用户可读状态。接口具体枚举值和状态变更时机需要统一标记为待黔云联创确认。

| 前端状态 | 用户展示 | 进入条件 | 页面操作 |
|---|---|---|---|
| `CREATING` | 正在提交订单 | 正在调用 `createOrder` | 禁止重复提交 |
| `WAIT_PAY` | 待支付 | 订单已创建但尚未完成支付 | 提供“继续支付”或“取消订单”；具体订单状态值待黔云联创确认 |
| `PAYING` | 支付处理中 | 已调用 `wechatPayment` 或 `wx.requestPayment` 尚未得到最终订单状态 | 显示处理中，不展示支付成功 |
| `PAID_ISSUING` | 支付成功，正在出票 | `getOrderByNo.orderStatus` 表示已支付但核销码尚未生成 | 可查看订单，提示稍后查看票码 |
| `ISSUED` | 支付成功 | 订单已支付且已出票/已有核销码 | 查看订单、查看核销码 |
| `PAY_FAILED` | 支付失败 | 支付接口明确失败且订单未支付 | 返回预订页重试；不可继续展示成功页 |
| `CANCELLED` | 订单已取消 | 用户取消支付或订单已关闭 | 返回商品详情；是否可重新支付待黔云联创确认 |
| `REFUNDING` | 退款处理中 | 订单进入退款流程 | 查看订单，不展示可用核销码 |
| `REFUNDED` | 已退款 | 退款完成 | 查看退款结果 |
| `UNKNOWN` | 订单状态确认中 | 接口返回未识别状态或查询失败 | 保留订单号，提示稍后重试；状态值待黔云联创确认 |

### 5.4 支付异常处理

| 场景 | 页面处理 | 订单处理 |
|---|---|---|
| 创建订单失败 | 留在预订弹层，展示错误提示 | 不进入支付结果页 |
| 微信支付参数获取失败 | 展示“支付未发起”，提供重试 | 通过 `getOrderByNo` 查询是否已产生支付结果 |
| 用户主动取消微信支付 | 不展示“支付成功”，保留订单号 | 查询订单状态；若仍待支付，允许重新支付 |
| 微信支付回调成功但出票未完成 | 展示“支付成功，正在出票” | 继续查询订单，不直接展示核销码 |
| 订单状态查询失败 | 展示“订单状态确认中” | 不把查询失败当作支付失败，支持重新查询 |
| 支付成功但订单关闭/退款 | 展示订单实际状态 | 以黔云联创订单状态为准，不允许进入核销流程 |

### 5.5 页面进入与离开规则

1. 支付结果页必须携带真实 `orderNo`，禁止只依赖前端内存中的订单金额和票型。
2. 从支付结果页点击“查看订单”，调用 `getOrderByNo`，并根据最新 `orderStatus` 刷新页面。
3. 支付结果页重新进入或小程序从后台恢复时，应重新查询订单状态。
4. 只有订单状态为已支付且出票完成时，才调用 `getResourcesOrderVoucher` 展示核销码。
5. 支付结果页当前原型可模拟成功；正式版本必须接入 `createOrder`、`wechatPayment`、`wx.requestPayment` 和 `getOrderByNo`。

## 7. 当前页面需要优先补齐的接口改造项

1. 接入 `getResourcesMoreInfo`，保存真实 `resourcesId`、`resourcesSkuId`、`checkType`、`hasRealName`。
2. 接入 `getResourcesStockAndPrice`，用真实日期、价格、库存和时段替换静态 mock。
3. 创建订单时按 `details[].number` 传数量，不要把前端 `quantity` 原样传给黔云联创。
4. 当前 MVP 固定证件类型 `cardTypeId=ID`；后续支持其他证件时再接入证件类型接口。
5. 当前临时使用梅花山景区近似固定坐标 `longitude=104.82`、`latitude=26.59`；待黔云联创确认是否替换为其商家详情返回的官方坐标。
6. `shopHours` 待黔云联创确认是否代表景区开闭园时间后，才能映射到“开放时间”。
7. “建议游玩”在黔云联创文档中没有对应字段，应由景区配置或独立内容接口提供。
8. 真实创建订单成功后，使用返回的 `orderId`、`orderNo`、`payAmount`、`orderStatus` 进入支付流程。
9. 支付流程为：创建订单 → 调用 `wechatPayment` → 获取 `payInfo` → 调用微信 `wx.requestPayment` → 查询订单状态 → 展示支付完成页。

## 9. 结论

当前页面已经覆盖了票型展示、数量选择、日期、时段、联系人和实名游客信息的主要交互，并已在 `src/services` 预留真实接口调用。未配置环境变量时使用静态 mock 数据；正式接入时仍需补齐真实资源/SKU ID、库存时段和订单明细组装，并完成待黔云联创确认项。

当前静态预览在完成本地校验后直接模拟支付成功，用于确认页面流程；真实小程序需要按本节顺序接入 `createOrder`、`wechatPayment`、`wx.requestPayment` 和 `getOrderByNo`，不能将模拟成功逻辑直接用于生产。
